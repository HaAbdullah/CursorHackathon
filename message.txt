## FAKE_LLM mode

When `FAKE_LLM=1`, every function in `extract.py` and `research.py` returns canned fixtures with zero network calls. Fixtures (exact):
- `extract_from_photos` and `extract_via_search` → `ExtractionResult(found_ingredients=True, product_name="VoltFuel berry drink", ingredients=["Carbonated water", "Sugar (27g)", "Caffeine (200mg)", "Taurine", "Ginseng extract", "Niacin", "Sucralose"])`
- research verdict for any unknown name → `IngredientVerdict(name=<normalized name>, risk_level="caution", why="Limited safety data during pregnancy.", condition_flags={})`, produced after `asyncio.sleep(4)` so the frontend loading state is exercised.

This mode is how the API and frontend run end-to-end before real LLM calls exist, and the on-stage fallback if a key dies.

## Database schema (`app/schema.sql`, verbatim)

SQLite. `db.py` must execute `PRAGMA journal_mode=WAL;` and open with `check_same_thread=False` (background task writes while requests read). All timestamps are ISO 8601 UTC strings. JSON columns are TEXT holding serialized JSON.

```sql
CREATE TABLE IF NOT EXISTS ingredients (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT UNIQUE NOT NULL,
  aliases TEXT NOT NULL DEFAULT '[]',
  risk_level TEXT NOT NULL CHECK (risk_level IN ('avoid','caution','safe')),
  why TEXT NOT NULL,
  max_amount TEXT,
  condition_flags TEXT NOT NULL DEFAULT '{}',
  source TEXT NOT NULL DEFAULT 'seed' CHECK (source IN ('seed','agent')),
  created_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS products (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  cache_key TEXT UNIQUE NOT NULL,
  name TEXT,
  barcode TEXT,
  ingredients_json TEXT NOT NULL,
  created_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS scans (
  id TEXT PRIMARY KEY,
  status TEXT NOT NULL CHECK (status IN ('processing','partial','complete','failed')),
  input_type TEXT NOT NULL CHECK (input_type IN ('barcode','photos')),
  product_name TEXT,
  ingredients_json TEXT NOT NULL DEFAULT '[]',
  profile_json TEXT NOT NULL,
  report_json TEXT,
  error TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

Column meanings: `ingredients.name` is canonical lowercase; `ingredients.aliases` is a JSON array of lowercase strings; `ingredients.condition_flags` is a JSON object whose keys come ONLY from the Condition enum values plus `trimester_1|trimester_2|trimester_3`; `products.ingredients_json` and `scans.ingredients_json` are JSON arrays of RAW ingredient strings exactly as extracted; `products.cache_key` is the barcode if one was provided, else the sha256 hex of the lowercased product name; `scans.id` is `uuid4().hex`.

Cache semantics: `products` stores only raw extracted strings — never verdicts, never personalization. A cache hit re-runs matching + personalization fresh (cheap, zero LLM calls), so cached products automatically pick up every verdict the agent has learned since.

## `app/models.py` (verbatim, frozen)

```python
from enum import Enum
from pydantic import BaseModel, Field
from typing import Optional


class RiskLevel(str, Enum):
    avoid = "avoid"
    caution = "caution"
    safe = "safe"


class IngredientStatus(str, Enum):
    avoid = "avoid"
    caution = "caution"
    safe = "safe"
    pending = "pending"


class ScanStatus(str, Enum):
    processing = "processing"
    partial = "partial"
    complete = "complete"
    failed = "failed"


class Condition(str, Enum):
    gestational_diabetes = "gestational_diabetes"
    hypertension = "hypertension"
    anemia = "anemia"
    thyroid = "thyroid"


class Profile(BaseModel):
    trimester: int = Field(ge=1, le=3, default=2)
    conditions: list[Condition] = []
    allergies: list[str] = []


class ExtractionResult(BaseModel):
    found_ingredients: bool
    product_name: Optional[str] = None
    ingredients: list[str] = []


class IngredientVerdict(BaseModel):
    name: str
    aliases: list[str] = []
    risk_level: RiskLevel
    why: str
    max_amount: Optional[str] = None
    condition_flags: dict[str, str] = {}


class MatchResult(BaseModel):
    raw: str
    canonical: str
    amount: Optional[str] = None
    verdict: Optional[IngredientVerdict] = None


class ReportIngredient(BaseModel):
    name: str
    raw: str
    status: IngredientStatus
    why: Optional[str] = None
    max_amount: Optional[str] = None
    personal_note: Optional[str] = None
    source: str


class Report(BaseModel):
    product_name: Optional[str]
    verdict: str
    summary: Optional[str] = None
    pending_count: int = 0
    ingredients: list[ReportIngredient] = []


class ScanResponse(BaseModel):
    scan_id: str
    status: ScanStatus
    report: Optional[Report] = None
    error: Optional[str] = None
```

`IngredientVerdict` deliberately mirrors an `ingredients` table row one-to-one: research output, DB rows, and match results all share one shape. Never introduce a second representation.

## Normalization and matching rules

`normalize(raw)`: 1) lowercase, strip whitespace; 2) extract an amount via regex `\(?\s*(\d+(?:\.\d+)?)\s*(mg|g|mcg|µg|ml|%)\s*\)?` into a human string like `"200 mg"` and remove it from the name; 3) strip a leading `contains`/`ingredients:` prefix and trailing punctuation; 4) return `(canonical_name, amount_or_none)`.

Match rule: canonical name matches a DB row iff `canonical == row.name` or `canonical in json.loads(row.aliases)`. Exact membership only — no fuzzy matching, no embeddings.

## HTTP API (the frontend contract)

CORS: allow all origins, methods, headers. All responses JSON.

`POST /scan` — multipart/form-data: `photos` (file, repeatable, 0–5, jpeg/png/webp), `barcode` (text, optional), `profile` (text, required — JSON string parsing as `Profile`). At least one of photos/barcode else HTTP 400 `{"detail": "provide photos or barcode"}`; bad profile → HTTP 400 `{"detail": "invalid profile"}`. Runs the synchronous path (lookup/extraction + matching), schedules background research if unknowns exist, responds:

```json
{ "scan_id": "9f3c2ab04e2d4d1b8f0a7c5e6d4b3a21", "status": "partial", "report": { ... } }
```

`status` is `complete` (no unknowns, or cache hit), `partial` (research running), or `failed` (nothing usable extracted — `report` null, `error` set). Extraction failure is HTTP 200 with `status:"failed"`, never a 5xx.

`GET /scan/{scan_id}` → `{"scan_id", "status", "report", "error"}`; unknown id → 404. Frontend polls every 2s while `partial`.

`GET /health` → `{"ok": true, "fake_llm": <bool>}`.

Report shape example:

```json
{
  "product_name": "VoltFuel berry drink",
  "verdict": "avoid",
  "summary": "With gestational diabetes, this drink's sugar and caffeine load make it one to skip.",
  "pending_count": 0,
  "ingredients": [
    { "name": "caffeine", "raw": "Caffeine (200mg)", "status": "avoid",
      "why": "One serving hits the full recommended daily pregnancy limit.",
      "max_amount": "200 mg/day", "personal_note": null, "source": "database" },
    { "name": "taurine", "raw": "Taurine", "status": "caution",
      "why": "Limited safety data at energy-drink doses during pregnancy.",
      "max_amount": null, "personal_note": null, "source": "agent" }
  ]
}
```

Field rules, no exceptions:
- `verdict` = worst ingredient status with order avoid > caution > safe, computed over non-pending ingredients only; if ALL are pending, `verdict` is `"pending"`.
- `summary` is null until scan status is `complete`.
- `pending_count` = count of ingredients with status `pending`; zero when complete.
- `ingredients[].source` ∈ `database` (matched an existing row), `agent` (researched during this scan), `pending` (still researching).
- `ingredients[].personal_note` = the matched row's `condition_flags[c]` for every condition `c` in the profile, plus the `trimester_{n}` key if present, joined with `"; "`; null if no overlap.
- `ingredients[].raw` preserves the extracted string exactly.
